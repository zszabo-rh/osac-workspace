Details
* Bare Metal Networking API Design and Implementation: Lars Kellogg-Stedman and Tzu-Mainn Chen discussed the bare metal networking API design and reached an implementation plan for the bare metal services. Lars Kellogg-Stedman will begin implementation work on the networking side and plans to review the networking API proposal today, following the meeting. While the implementation will likely not be finished by Monday, the team has a clear idea of what the networking API should look like.
* Networking API Scope and Multi-Implementation Support: The bare metal implementation of the networking API is based on the existing enhancement proposal and is expected to support multiple networking implementations. However, the current assumption is that the networking implementation is closely tied to the underlying bare metal environment, meaning that it is not trivial to connect devices across different networking implementations. This assumption is based on the idea that networks created are typically private to the environment in which they are created.
* Necessary Adjustments to the Networking API: Eran Cohen noted that the current networking API, which started as an API for VM as a service, is likely to require adjustments to fully apply to bare metal as a service. This will be the first version, and they expect changes to occur once the implementation details are finalized. Lars Kellogg-Stedman also raised a specific need for a network attachment resource within the API to declare a binding between a device, such as a bare metal node, and a network.
* Bare Metal Network Interface Scope: Orran Krieger questioned whether the bare metal implementation would use the full networking API or a subset focusing on Layer 2 capabilities. Lars Kellogg-Stedman confirmed that while the initial focus is on Layer 2 networks and machine attachment, the team ultimately expects to support the complete network API, enabling Layer 3 actions on bare metal machines. The assumption is that different network stacks will be effectively isolated, connecting only at routed levels.
* Inventory and Network Management Tool Relationship: Michael Hrivnak suggested that the tooling used for fabric management should be considered separate from the inventory of existing resources, noting that tools like Ansible and Netbox exemplify this separation. Lars Kellogg-Stedman and Tzu-Mainn Chen clarified that the current working assumption is a one-to-one relationship, where each bare metal inventory is managed by only one network management tool, though they are not assuming a tight coupling between the tools themselves. They acknowledged that implementation details will become clearer as work progresses.
* Timeline and Drivers for Bare Metal Work: Heidi Dempsey inquired about the timeline, especially the mention of a Monday deadline, to understand the drivers behind the urgency. Lars Kellogg-Stedman clarified that the Monday reference was based on a previous question from Alona and that they are not under pressure from a particular deployment but are moving forward because the bare metal capability is necessary. The immediate plan is to define the interface and create a plugin for OpenStack first, with the ultimate goal of integrating with external infrastructure and the cluster-as-a-service functionality.
* Bare Metal Allocation for MLC 2.0: The team discussed the deployment of the bare metal network management capability for the MLC 2.0 environment, which is starting soon. Heidi Dempsey asserted that static allocation would be necessary for the initial phase of the MLC 2.0 deployment, as the new service will not be ready in time. The goal is to integrate the service as a phase-in later, allowing for elastic resource allocation after the initial static setup.
* Company Buy-in on Sovereignty and OSAC: Michael Hrivnak reported that discussions about the project's direction and investment have gone well, exceeding the goals for the first phase. The company has shown strong and continued buy-in on the narrative surrounding sovereignty and cloud patterns, with OSAC becoming an increasingly important path for this work.
* Sovereign Cost Management Solution: Christopher Tate introduced Red Hat's new sovereign cost management solution, which operates without calling back to redhat.com and is currently in early development. The MOC is collaborating with the Red Hat team on this solution to incorporate MOC-specific cost management needs, particularly around GPU utilization tracking based on allocated resources rather than utilization metrics. This product aims to replace the custom cost management solutions currently used by the MOC.
* Cost Management Product Integration and Policy: Heidi Dempsey confirmed a meeting is scheduled to discuss the new sovereign cost management solution in detail, emphasizing the value of replacing custom code with a supported product if it meets their needs. Orran Krieger raised questions about the product's policy flexibility, and Christopher Tate confirmed that it supports service unit bundling and will implement allocation-based billing. Lars Kellogg-Stedman suggested exploring whether the associated policy work for billing is managed by the ACM product or a separate system, as ACM policy often focuses on cluster configuration.
* Exposing Cost Information in OSAC: The team agreed that OSAC needs a way to expose utilization and export metric values for billing, but Lars Kellogg-Stedman stressed that the cost logic itself should reside outside of OSAC. Orran Krieger highlighted the importance of having an interface to an external system to coherently display cost information to users, similar to other cloud platforms.
