Summary
The team debated the bare metal host assignment architecture, agreed on the bare metal operator's role, and reviewed open enhancements.

Bare Metal Proposal Consensus
The team confirmed general agreement on the bare metal proposal, but the core disagreement persisted over whether the bare metal operator or the host operator should handle inventory interaction. Participants ultimately agreed that the bare metal operator must set the management class on the Host CR to avoid a “chicken and egg problem.”

Enhancement and PR Review
The review of 11 open enhancements led to the closure of several superseded or erroneous proposals, including the quota and serial console enhancements. The team reviewed open pull requests, with Eran Cohen and Lars Kellogg-Stedman volunteering to look at a large Ansible code reorganization PR.

Networking Implementation Status
Eran Cohen confirmed that basic networking functionalities are working, but the critical next steps include adding network attachment functionality to the compute instance and implementing security groups. The team discussed the need for network abstractions that support parallelism and do not overcomplicate integrations like Netris.


Details
* Proposal Discussion and Enhancement Conversion: The team discussed the bare metal proposal, confirming general agreement from a previous Thursday meeting, but noted a sticking point regarding where host assignment should be performed. Tzu-Mainn Chen prefers keeping the proposal in a Google Doc for broader discussion before converting it to an enhancement proposal on GitHub (00:00:00). Alona Paz agreed to proceed with the Google Doc review (00:01:11).
* Disagreement on Host Assignment Component: The core disagreement centers on whether the bare metal operator or the host operator is responsible for interacting with the inventory (00:01:11). Lars Kellogg-Stedman asserts that the bare metal operator should interact with the inventory. They explained that the bare metal operator is the higher-level component that receives host requests, while the host operator reconciles Host Custom Resources (CRs) and communicates with the bare metal management back end (00:02:15).
* Argument for Bare Metal Operator Handling Inventory: The argument is that the bare metal operator must interact with the inventory before creating a host CR because it needs to determine the management class (e.g., Ironic or Carbide) required for reconciliation. The bare metal operator should specify an appropriate management class on the host CR so the correct host operator can pick it up; otherwise, a "chicken and egg problem" arises if the host operator tries to do the inventory interaction (00:03:21).
* Counterargument and Kubernetes Conventions: Michael Hrivnak's perspective, as summarized by Lars Kellogg-Stedman, favored the host operator handling inventory because it is common in Kubernetes for a single operator to reconcile a resource in its entirety (00:04:37). Eran Cohen questioned the proposed model where different host operators reconcile the same CR, but Lars Kellogg-Stedman likened this approach to Kubernetes storage classes, where different subsystems manage classes based on criteria set in the resource (00:05:42).
* Conclusion and Follow-up on Host Assignment: The participants generally agreed with the proposed model where the bare metal operator sets the management class, aligning it with standard Kubernetes practices like storage classes (00:05:42). Lars Kellogg-Stedman and Tzu-Mainn Chen plan to proceed with their intended design but will follow up with Michael Hrivnak to see if they can convince them otherwise (00:06:39). Tzu-Mainn Chen will follow up in Slack with Michael Hrivnak regarding the conversation (00:07:36).
* Acceleration and Coordination of Development Efforts: Orran Krieger suggested accelerating development by involving other people, specifically from Ecosystem Engineering, given the need for multiple backends (00:07:36). Tzu-Mainn Chen confirmed that coordination is already happening in a bare metal meeting that Trey from Carbide attends. Alona Paz requested that Dan Manor be added to the bare metal meeting because they are working on metrics that will require inventory and host management (00:08:27).
* Moving Forward with Proposal and Implementation: Alona Paz summarized that the team agrees with the proposed model and plans to seek final agreement from Michael Hrivnak. Once the proposal is settled, it will be moved to a GitHub enhancement proposal to initiate implementation (00:08:27). Alona Paz also mentioned that the work should be broken down into tasks, and Ecosystem Engineering personnel will be assigned to assist (00:09:47).
* Discussion with Michael Hrivnak on Inventory: Michael Hrivnak joined the meeting, and Lars Kellogg-Stedman reiterated the assertion that the bare metal operator must handle the inventory interaction to determine the correct management backend before creating the host CR (00:10:51). Michael Hrivnak questioned why the bare metal operator must be the *only* component to talk to the inventory, but Lars Kellogg-Stedman explained that in an environment with multiple backends, the inventory is the source of truth for which backend manages a particular host (00:11:54).
* Defining Host Intent and Acquisition Process: Michael Hrivnak inquired about the intent used when requesting hosts, which Lars Kellogg-Stedman clarified includes match expressions on host attributes like model, GPU, or memory (00:14:04). Lars Kellogg-Stedman described the proposed process: the bare metal operator receives the request, contacts the inventory, selects and marks hosts as assigned, and then creates the host CRs with the appropriate management class for the host operator to reconcile (00:15:23).
* Alternative Model for Management Class Determination: Michael Hrivnak suggested an alternative where the host CR is created with intent, and a host controller finds a matching host in the inventory and sets the management class (00:16:28). This led to a discussion of how multiple host operators would then know who gets to reconcile the CR, which Lars Kellogg-Stedman argued is solved more easily by setting the management class at the higher bare metal operator level (00:17:54).
* Review of Open Enhancements and Closures: Alona Paz led a review of 11 open enhancements, starting with old ones for possible retirement. The quota enhancement was closed as superseded, and the bare metal observability enhancement was closed temporarily since Austin Jamias is working on the bare metal cluster operator (00:26:21). The enhancement for adding SLURM on OpenShift remains blocked pending feedback from Michael Hrivnak (00:27:43).
* Enhancements Actioned and Reviewed: The Dependabot enhancement was merged, and the serial console enhancement proposal was closed because it was created in error and was not related to Zultan's implementation (00:30:18). The feature tracking storage class was noted as already implemented and close to being merged (00:31:18). Lars Kellogg-Stedman volunteered to review the new Carbide and quota enhancements. Kristi Nikolla also confirmed they would review the quota enhancement, and Akshay Nadkarni confirmed that the implementation for the tenant storage class is underway (00:32:30).
* Review of Open Pull Requests (PRs): The team reviewed open pull requests, and Eran Cohen volunteered to review a large PR focused on reorganizing Ansible code, with Lars Kellogg-Stedman also volunteering to look at the Ansible PR (00:35:01). The three open networking PRs were confirmed as already reviewed and merged (00:36:05).
* Networking Implementation Status and Next Steps: Eran Cohen reported that basic networking, including CUDN, generic API, network class, virtual network, and subnets, is working, though the implementation of security groups is partial. The next critical step is adding network attachment functionality to the compute instance and security groups, as well as registering network classes (00:36:05). Lars Kellogg-Stedman agreed to review the current implementation to ensure their bare metal plans for networks and network attachment resources align (00:37:28).
* Discussion on VLANs and Network Abstractions: Heidi Dempsey raised the topic of VLAN management, and Lars Kellogg-Stedman clarified that the adopted OpenStack terminology uses "network" to mean a layer 2 segment, typically an isolated VLAN in their bare metal environment, and tenants do not directly manage VLAN IDs (00:38:58). Eran Cohen confirmed that the network class helps determine whether a virtual network is a VLAN or another abstraction based on the environment (00:41:14).
* Networking Requirements for Bare Metal: Lars Kellogg-Stedman specified that bare metal requires networks and network attachment resources to specify connectivity between hosts and networks (00:37:28) (00:41:14). Orran Krieger emphasized that for physical hosts, the abstraction needs to account for which physical network interface card (NIC) is connected to which switch port (00:42:01). Lars Kellogg-Stedman clarified that the network attachment resource is a container for the information needed to link a network to a particular NIC on a host (00:44:14).
* Concerns about Networking API Complexity (Netris): Dan Manor expressed concern that the new central networking API might over-complicate things for integrations like Netris, which currently automates many network provisioning steps (VPC, subnets, IP allocation) through a single "server cluster" concept (00:45:20). Orran Krieger supported the idea of designing a high-level, parallel abstraction that can handle batch operations, recognizing that serializing network operations limits concurrency (00:48:47).
* Conclusion on Batch Operations and API Scope: Lars Kellogg-Stedman acknowledged the goal of batch operations but noted that the current API design focuses on single machine allocation and does not natively support bulk operations (00:51:11). Dan Manor agreed and stated they did not want to change plans but proposed considering a higher-scope API for future efficiency. Orran Krieger stressed that for physical machines, the API should reflect that resources already exist and should aim for concurrency by utilizing vector-based, bulk operations (00:52:28) (00:56:11).
* Discussion on enabling new behavior: The participants acknowledged that enabling certain behaviors, such as the one referenced by Eran Cohen, is a long-term consideration. Lars Kellogg-Stedman suggested that this requires deeper thought about how to facilitate this kind of functionality. However, Heidi Dempsey emphasized that this planning should not impede the immediate deployment of current tasks (00:57:07).
* Network API Review Assignment: Due to time constraints, Alona Paz requested that Dan Manor review the network API. Specifically, they asked him to investigate how the API maps to the network, nodes, and what they map, followed by a discussion with Orran Krieger and potentially Lars Kellogg-Stedman (00:57:07).
* Emphasis on Parallelism in Interfaces: Orran Krieger discussed the importance of incorporating parallelism into the interface design. They asserted that expressing processes as parallel is always beneficial because it allows for serialization underneath the interface if necessary, while neglecting to express parallelism prevents its exploitation. Orran Krieger concluded that leveraging existing proper parallel interface classes is a valuable opportunity (00:57:07).
* Meeting Conclusion and Adjournment: The meeting concluded as the participants ran out of time. Alona Paz confirmed the time constraint and thanked the participants for attending. Dan Manor and Lars Kellogg-Stedman also offered their thanks before the meeting adjourned (00:57:07).
