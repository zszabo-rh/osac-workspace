Summary
Cluster-as-a-service flow was demonstrated with Netris integration, bare metal operator structure preferences were discussed, and a major review of advanced AI coding agents and spec-driven workflows was conducted.

Cluster-as-a-Service Demo
A recorded demonstration of the cluster-as-a-service deployment with Netris and Metal LB was presented, showing the cluster creation and network topology via the Netris UI. Discussion confirmed the development of a unified networking API based on the Adrian enhancement to abstract various backends like Neutron and Netris.

Bare Metal Operator Structure
An update on bare metal work indicated 2 CRDs were submitted, with 1 already merged. The group agreed on a slight preference for splitting backend-specific host management into separate, optionally installable operators rather than maintaining a single monolithic operator.

Review of AI Coding Agents
A deep dive into advanced AI coding agents highlighted the utility of spec-driven workflows, the GSD workflow engine, and the Claude 3 Opus model for maintaining context in implementation. The team discussed the need to formalize AI development practices, including documenting repository policies in a cloud.MD file, and integrating project management into tool adoption.


Details
* Initial Agenda Check and Topics: The meeting began without a previously circulated agenda due to Alona's absence, but Michael Hrivnak confirmed that three items were on the agenda. Dan Manor announced two topics for discussion: a short demo of the cluster-as-a-service flow with Netris, and discussion on finalizing changes and integrating them into the OSAP repository, specifically addressing issues with making the repository configurable for end-users (00:00:00).
* Demo of Cluster-as-a-Service Flow with Netris: Dan Manor presented a recorded demo of a completed cluster-as-a-service deployment, explaining that the operation was too long for a live demonstration. The setup included an SNO hosting cluster, with OSAC and its components deployed alongside Metal LB, which is currently required for the cluster-as-a-service implementation (00:01:55) (00:08:44). The demonstration showed a generic OCP hosted template that supports ESI, OpenStack Neutron, and Netris backend network classes, successfully creating a cluster with two nodes and available cluster operators (00:03:21).
* Networking Topology and Netris Integration: The Netris UI was shown to illustrate the network topology, including the north-south fabric and servers for agents and the management cluster communicating via the data fabric using VPCs (00:05:24). Dan Manor explained that the server cluster is a wrapper for networking tasks, and that future implementation will split these functionalities among different network endpoints (subnet, NAT, etc.) as the network class is implemented (00:07:04). The Netris API is invoked automatically during the cluster fulfillment process, similar to the current OpenStack Neutron and DSI implementations (00:08:44).
* Distinction Between Bare Metal and Networking Efforts: Lars Kellogg-Stedman clarified that the work being done by Austin, Danny, and Min focuses on bare metal hardware management (power control, provisioning, and allocation), and is distinct from the network configuration aspects that rely on components like Netris or Neutron, which Dan Manor presented (00:09:58). Tzu-Mainn Chen asked if all the networking work would eventually be placed behind a unified networking API (00:11:29).
* Future Network API Development and Metal3 Overlap: Dan Manor confirmed ongoing work, possibly by Eran Cohen, to create a new API for networking based on the Adrian enhancement, with the next step being to fulfill each of the new network API endpoints. Eran Cohen noted that there might be overlap with the Metal3 team, as they are reportedly adding network capabilities, though probably not supporting the full range of backends, and a meeting was scheduled to discuss this overlap (00:11:29). Orran Krieger views their team's effort as the layer agnostic to implementation, which can connect to Netris, OpenStack networking, or other proprietary solutions, hoping Metal3 could serve as a fully open-source solution for network management (00:13:04).
* Next Steps for Network Implementation: Tzu-Mainn Chen summarized the next priority for the networking process as getting the network API up and running with Neutron and Netris backends, presuming the current work gets merged, to make the flow cleaner. Lars Kellogg-Stedman then transitioned the discussion to the bare metal update (00:15:29).
* Bare Metal Update and Operator Structure: Tzu-Mainn Chen provided an update on the bare metal work, stating that a pull request for two CRDs was submitted last week, with one already merged and the other awaiting re-review from Michael Hrivnak (00:15:29). They asked whether it is preferable to have separate operators for backend-specific host management (like OpenStack or Carbide) or to maintain a single monolithic operator. Michael Hrivnak and Lars Kellogg-Stedman both expressed a slight preference for splitting out the backend-specific host management into separate operators that could be installed optionally based on local infrastructure (00:16:18).
* Discussion on AI Coding Agents and Workflows: Eran Cohen started the discussion on AI workflows, noting the significant improvement in AI coding agents, specifically mentioning using Cursor and the updated Claude (00:17:43). Eran Cohen highlighted the utility of the "get s*** done" (GSD) workflow engine for agentic development, which is spec-driven and focuses on executing the requested task without unnecessary deviations (00:19:22).
* Key Features and Tools for AI Development: Eran Cohen highlighted new features such as the Claude 4.6 Opus model with a 1 million context window, which is useful for maintaining context during implementation (00:19:22). They mentioned using GSD for spec-driven development but noted that simply triggering 'plan mode' in Claude without GSD provides a good level of planning and verification (00:20:53). A key capability mentioned is the ability to use the frontier Claude 4.6 model without apparent budget limits (00:22:17).
* Using Claude MD and Skills for Customization: The `cloud.MD` file in the repository root acts as a source of truth for Claude to understand how to operate in the repository (00:22:17). Eran Cohen uses "Skills," which are ways to harness Claude to perform tasks in a specific manner, with GSD being a set of such skills that simplify the process (00:23:38). Eran Cohen created plugins for handling external tools like Jira, GitHub, and Slack, which can be wrapped with a skill to automate complex, multi-step operations (e.g., fixing a bug, opening a ticket, running tests, and posting a PR) (00:24:54).
* AI for Enhanced Workflow Automation and Quality: Eran Cohen demonstrated how Claude can record demos, upload them, and send emails, showing its high capability once skills are integrated. Working in a spec-driven workflow with GSD provides reliable results and facilitates a comprehensive planning phase, where the agent asks challenging questions to refine the task scope, which is particularly helpful for large, complex tasks like the networking stack (00:27:56) (00:39:06).
* Benefits of Git Work Trees and Command Creation: Elad Tabak commented on the usefulness of Git work trees for mapping branches to different file system folders, which allows multiple agents or individuals to work in parallel on the same repository (00:31:08). Elad Tabak also demonstrated creating a custom `daily update` command for Claude to summarize their daily status into a file, which is then processed and posted to Slack via automation (00:32:37).
* AI for Reviews and Process Standardization: Avishay Traeger shared their positive experience using Claude for an enhancement proposal review, where Claude fetched the necessary information, facilitated a back-and-forth discussion on concerns, and formalized the resulting comments for posting (00:35:23). There was a discussion about whether to formalize a standard way of working with AI tools, with Eran Cohen and others suggesting they should recommend a default flow, such as having a `cloud.MD` file that describes repository policies and structure (00:44:41).
* The Debate on Standardization and Community Involvement: Orran Krieger suggested that AI development practices should be integrated into the project's repositories, not kept as individual efforts, and that the team needs to decide the extent of human involvement in each development step (00:40:14). Heidi Dempsey cautioned that any requirements must be open source and free, as OSAC is a community project. Orran Krieger agreed but emphasized the need for visibility into their internal AI practices and potentially supporting models for external, non-funded groups to ensure community members can use the same tools (00:42:36).
* Project Management and AI Tool Adoption: Heidi Dempsey stressed the need to include project management as part of the discussion on AI tooling adoption, particularly for measuring results, as they still lack a unified project management approach for OSAC (00:46:00). Orran Krieger agreed, noting that documenting their architecture and standards in the repository is crucial for consistency and to enable these tools to function properly (00:47:53).
* Call for Broader Adoption and Feedback: Oved Ourfali encouraged all participants to try out the shared tools, noting the industry trend toward agentic workflows, and emphasized that the process will be gradual, requiring shared opinions and feedback to evolve the toolset (00:49:52). Orran Krieger suggested adding examples and principles for using AI into the OpenShift repository documentation (00:51:29).
* Finalizing Discussions and Next Steps: Dan Manor requested to continue the discussion regarding the configuration of OSAP on Slack, as time was running out. Orran Krieger asked Eran Cohen to review the open shell example for documenting AI use in repositories to inform their own version (00:51:29).
