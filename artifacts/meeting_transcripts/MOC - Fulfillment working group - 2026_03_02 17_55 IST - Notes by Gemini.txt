Summary
Alona Paz introduced Dan Manor, who presented Netris as an automation platform for data center environments, outlining its value proposition for OSAC Cluster as a Service through features like multi-tenant isolation, automated network provisioning via VPCs, and integrated load balancing demonstrated in a proof of concept. The participants, including Orran Krieger and Michael Hrivnak, generally agreed that Netris is a compelling solution that aligns well with the OSAC design, potentially serving as an effective ESI replacement for greenfield deployments. The immediate next steps involve Dan Manor defining a new 'Netris' network class based on the existing networking API proposal and Lars Kellogg-Stedman prioritizing the implementation of the ESI network class as the first concrete example of the new networking API interaction, contingent on the core networking API pull requests being merged.


Details
* Introduction of New Team Member and Netris Integration Discussion: Alona Paz introduced Dan Manor, who will be joining the effort to integrate Netris into OSAC's Cluster as a Service. The initial work involved a proof of concept (PoC) with Netris, and the next phase is to discuss how to integrate it into OSAC, starting with service clustering. Dan Manor was prepared to present slides about Netris and the PoC they had completed.
* Netris Overview and Value Proposition for OSAC: Dan Manor presented Netris as an automation and fabric control platform for data center environments, exposing both Kubernetes and restful APIs to provide functionalities like VPCs, EVPN, VXLAN-based tenant isolation, distributed routing, integrated load balancing, and fabric-level ACL enforcement. For OSAC Cluster as a Service, Netris could address core fulfillment components in a multi-tenant environment, including tenant isolation, automated network provisioning (e.g., per-cluster subnet on-demand VPC creation), and connectivity between different VPCs.
* Proof of Concept Goals and Environment Setup: The PoC aimed to use Netris to configure networking for a multi-tenant Cluster as a Service environment, which included a hosting cluster running MCE and Hypershift, along with multiple tenant OpenShift clusters. Key objectives included ensuring tenant-to-management connectivity (where tenant worker nodes initiate communication), maintaining tenant isolation, providing external exposure via dedicated public IPs, and implementing load balancing for highly available management clusters. The minimal PoC used an emulated environment with a management cluster in a dedicated Netris VPC and two hosted clusters, each in their own VPC, with VPC peering configured between the tenants and management but not between the tenants.
* Netris Architecture and Deployment Details: Responding to a question about Netris components, Dan Manor explained that Netris primarily consists of a controller and agents, with agents running on switches and the controller running on the management or hosting cluster, likely deployed as a Kubernetes deployment. Orran Krieger added that Netris supports both Infiniband and Ethernet and has integrations with hypervisors and DPUs to achieve strong isolation. Orran Krieger noted that Netris is designed for greenfield deployments and does not assume integration with legacy enterprise environments.
* Netris Live Console Demonstration: Dan Manor showed the Netris console, which displayed the emulated environment's topology, including spine-leaf architectures for east-west and north-south fabrics, and servers meant to be GPU servers. They explained that Netris manages load balancing and VPC pairing, which allows for communication between the hosted clusters and the management cluster while restricting direct tenant-to-tenant communication. The communication limitations were enforced using ACL rules.
* Netris Configuration Mechanism and Alignment with OSAC Design: Dan Manor explained that when provisioning a server cluster, Netris configures the network on the switches, handling the virtual network setup in the background. Orran Krieger noted that Netris's use of machine templates and an inventory system (with strong demand for integration with NetBox) aligns well with OSAC's Cluster as a Service design, potentially serving as a proprietary, effective plugin for large, greenfield deployments, similar to what was envisioned for an ESI replacement. Michael Hrivnak agreed that Netris is compelling and solves problems well, suggesting that it should be treated as a solution from a primary partner, similar to how other third-party vendors are approached, without locking into a single provider.
* Discussion on Netris, NetBox, and Integration Strategy: The team discussed the need for further exploration of Netris's integration with NetBox, which Orran Krieger noted was a high customer demand for Netris, though they currently use their own inventory system. Alona Paz emphasized focusing on integrating the networking part, rather than tackling the inventory issue simultaneously, specifically suggesting working on the ESI replacement aspect. Rom Freiman and Michael Hrivnak debated the correlation between the network provider and the source of truth for the server inventory.
* Next Steps: Integrating Netris into Cluster as a Service via New Networking API: The consensus was to focus on integrating Netris into the Cluster as a Service story by first exploring the Netris APIs and mapping them to the APIs being developed internally. Eran Cohen and Lars Kellogg-Stedman agreed that, given the lack of a defined networking story for VM as a Service, the focus should remain on Cluster as a Service. Alona Paz proposed proceeding with a new network class, potentially named 'Netris', that aligns with the existing networking proposal being implemented by Adrian Gentil.
* Motivation and Need for a Real Environment Sandbox: Lars Kellogg-Stedman inquired about the motivation for prioritizing Netris, and Alona Paz confirmed it is a high-priority request driven by the Foxconn initiative and general customer demand, leading Netris to join Red Hat's partnership program. Orran Krieger suggested establishing a real environment sandbox with Netris at the MOSC to move beyond the emulated environment, noting that this would require acquiring or dedicating hardware (specifically layer 3 switches) that would not disrupt existing GPU infrastructure. Lars Kellogg-Stedman and Orran Krieger concluded that acquiring a new switch for a sandbox would be the preferred, non-disruptive method, although acquiring the hardware could take weeks.
* Addressing ESI Replacement and Bare Metal Control: Alona Paz asked what would be missing if ESI were eliminated from the Cluster as a Service story, assuming Netris handles the networking. Lars Kellogg-Stedman identified the missing components as the inventory and bare metal control aspects. Juan Hernández clarified that in the Foxconn PoC, they took inventory details like BMC credentials from an external system (BCM) and used ACM/MetalKube for power operations. Lars Kellogg-Stedman and Michael Hrivnak suggested that Bare Metal Hosts could serve as a short-term inventory solution, though they agreed a common identifier would be necessary to bridge the gap between this inventory and Netris.
* Immediate Plan: Implementing ESI Network Class and Reviewing API PRs: The team decided that the immediate plan is for Dan Manor to work on defining the new network class for Netris using the existing API proposal. Lars Kellogg-Stedman was asked to prioritize implementing the ESI network class, which would serve as the first concrete example of how Cluster as a Service interacts with the new networking API. The plan requires the core networking API PRs to be merged, and the participants were urged to review them, as they define the foundational networking object and implementation details like how to pass implementation-specific parameters via annotations. Lars Kellogg-Stedman accepted the task of prioritizing the ESI network class implementation.
* Review of Pull Request and Merger Priority: Alona Paz confirmed that Lars Kellogg-Stedman would review a pull request, as they are involved in implementing the cluster-as-a-service component. Eran Cohen and Adrien Gentil had previously offered positive feedback on the request. Alona Paz stated that this review is the first priority and expressed hope that the pull request would be merged quickly.
* Discussion on Infrastructure and Intent Validation: Orran Krieger raised questions about where dealing with physical infrastructure fits into the current thinking, citing past work on ESI and a company called nrris.io that manages switches. They also inquired about how the intent is validated to ensure that actual execution is consistent with the initial intent. Orran Krieger mentioned that they are dealing with "too much" activity currently.
* Compliance and External Auditing for Infrastructure: Orran Krieger connected the infrastructure topic to compliance requirements, referencing the N 800 171 standard. They noted that one cannot self-apply for compliance; instead, developing runbooks and mechanisms only meets the requirements if an external auditor validates the deployment. Orran Krieger suggested that collaborating on compliance testing might be beneficial, especially due to their need to meet compliance regimes as part of an RPA initiative, which could lead to a public validation statement.
