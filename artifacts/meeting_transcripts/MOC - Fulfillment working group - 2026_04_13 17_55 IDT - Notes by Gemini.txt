Summary
Documentation strategy finalized via component-specific repository inclusion with bare metal fulfillment proposal approval and initial AI policy drafting.

CLI and Documentation Strategy
The group agreed to rename the fulfillment CLI to OSAC CLI for better branding and approved a documentation strategy mirroring the hypershift pattern. Component-specific documentation, like the OSAC operator, will reside within its repository while the docs monorepo aggregates content.

Bare Metal Fulfillment Approval
The updated proposal for bare metal fulfillment received approval and was prioritized for immediate merging to support MSA 2.0 implementation. This proposal included the decision to separate the bare metal fulfillment operator from the main OSAC operator for easier release management.

Initial AI Policy Review
A draft AI policy was presented emphasizing human responsibility for all contributions and the requirement for code to be open source and human-reviewable via PRs. The group agreed to finalize the policy within 24 hours and move it into a PR in the documentation repository as a high-level guidance document.

Details
* Planned Absence and CLI Renaming: Lars Kellogg-Stedman announced they will be out next Monday. Will Gordon proposed renaming the `fulfillment CLI` to simply `OSAC CLI` or `OSAC` due to the length of the current name, which was met with consensus and deemed good branding (00:00:00). Will Gordon committed to submitting a pull request (PR) for this renaming (00:01:07).
* Documentation Strategy Discussion: Will Gordon proposed a documentation strategy where documentation for specific components, such as `OSAC installer`, `OSAC operator`, `AAP`, and `fulfillment service`, would be located within their respective repositories, similar to the hypershift pattern (00:01:07). The existing docs monorepo would then aggregate and publish the documentation, which Michael Hrivnak and Avishay Traeger supported for maintaining the cohesiveness of the overall OSAC story and keeping technical documentation close to the code (00:03:19). Product documentation remains uncertain until a product path is decided, which will heavily influence the eventual publishing location (00:02:20).
* Centralized vs. Component-Specific Documentation: The group agreed that the high-level architecture, overall API, and getting-started guides should remain centralized in the docs monorepo (00:03:19). However, component-specific or technical documentation, such as CRD references for the OSAC operator, should be generated in the component’s repo and aggregated, ensuring the documentation stays current with the code (00:04:19). Lars Kellogg-Stedman endorsed this approach, noting it prevents documentation from lagging behind code development (00:05:48).
* Documentation Aggregation Mechanism: Will Gordon proposed using GitHub actions to trigger the docs monorepo to pull updates from component-specific repos upon a PR merge or via a scheduled timer (00:07:54). This mechanism would enable the docs repo to function as the aggregator, pulling and publishing the content to an online source like Netlify or Read the Docs (00:05:48) (00:08:56). Michael Hrivnak suggested that the CLI documentation could serve as the initial proof point for setting up this automation, with Avishay Traeger noting the need for both CLI reference materials and use case-based documentation (00:10:02).
* Bare Metal Fulfillment Proposal: Tzu-Mainn Chen presented an updated proposal for bare metal fulfillment, indicating that Dimitri had reviewed and approved the overall philosophy and the bare metal host CRD. Tzu-Mainn Chen expressed urgency for merging this proposal to support MSA 2.0 implementation and requested any remaining major objections be raised soon. Lars Kellogg-Stedman planned to review and likely approve the proposal after lunch (00:11:56).
* Separation of Bare Metal Fulfillment Operator: Tzu-Mainn Chen revisited the topic of separating the bare metal fulfillment operator from the main OSAC operator, suggesting this separation could be beneficial for release management and managing entwined code. Lars Kellogg-Stedman generally favored the separation, noting it simplifies development and deployment and is an easily revisitable decision (00:12:54). Michael Hrivnak agreed that the independent manageability of bare metal, which could be used outside of OSAC, justifies its separation from the OSAC operator (00:15:47).
* Renaming of Host Class to Host Type: Tzu-Mainn Chen proposed renaming the concept of `host class` to `host type` to better align with existing terminology like instance type in the virtualization space. Under this proposal, `host type` would refer to the hardware categorization (e.g., H100), while the current host class concept would be reserved for the management backend (e.g., ironic or carbide), aligning well with terms like ingress class and storage class (00:17:04). This proposal received agreement from the participants (00:18:05).
* AI Policy Discussion: Eran Cohen presented a draft AI policy, emphasizing that the human remains responsible for all contributions and that all code should be open source, pass testing, and be reviewable by humans through PRs. The policy, which received significant feedback, does not define the vision of the agentic development lifecycle but sets initial ground rules (00:19:08). Orran Krieger praised the policy as a good start, suggesting that the next step should involve documenting best practices for using AI in the repositories and evolving them collaboratively (00:20:33).
* AI Policy Resolution and Submission: Heidi Dempsey noted that the policy needs to be moved out of the Google Doc and into a PR in the docs repository as a high-level policy document (00:21:26). Eran Cohen agreed to ask those with unresolved comments to finalize them within 24 hours, after which they would submit the current document as a PR (00:23:51). Lars Kellogg-Stedman confirmed they were happy with the policy as written (00:23:06).
