Summary
Roadmap discussion emphasized complete demos for finalized features, spurring a detailed debate on Cluster as a Service storage implementation and its necessary API alignment with VM as a Service.

Roadmap and Demo Policy
The OSAC roadmap includes epics for cluster and bare metal services, with an emphasis that demos should only occur when an epic is fully completed and merged. There was discomfort expressed about relying heavily on the Jira calendar for demo scheduling, suggesting a need for a more definitive completion metric.

Cluster Storage Strategy Review
Storage for Cluster as a Service lacks definition, prompting discussion of options like using a provider's Container Storage Interface (CSI) driver or implementing an OSAC CSI driver for centralized quota management. It was decided that determining if the Cluster as a Service storage API should be shared or separate from the VM as a Service API is urgent.

GPU Instance Types and Automation
Demos showcased VM provisioning with networking and GPU pass-through, leading to concerns that creating new Ansible templates for every GPU type requires excessive administrative effort. Discussion concluded that instance types must be defined consistently across bare metal and VMs, requiring automated configuration to ease the cloud administrator’s workload.

Details
* OSAC Roadmap and Epic Management: Alona Paz introduced a document for the OSC road map that includes epics from Jira for cluster as a service and bare metal as a service. They requested that all individuals working on an epic review the document and notify them if the proposed demo date is inaccurate or if a date needs to be added (00:00:00). Alona Paz emphasized that demos should only occur when an epic is fully complete, merged, and the feature is ready to be presented (00:01:20).
* Discussion on Storage for Cluster as a Service: The discussion moved to storage, focusing on generic storage API, multi-tier storage, and tenant onboarding for storage class, which are primarily related to VM as a service. Alona Paz inquired if there were plans to extend the API to cluster as a service. Avishay Traeger and Michael Hrivnak confirmed that little thought has been put into cluster as a service storage, with Tzu-Mainn Chen indicating that Lars, who was absent, would be the best person to address the topic (00:02:29).
* Storage Implementation Options for Cluster as a Service: Avishay Traeger outlined two options for cluster storage: using the provider CSI driver (e.g., install a Pure Storage CSI driver) or implementing an OSAC CSI driver to route everything through the OSAC API for quotas and billing. Will Gordon suggested a third option using a validating admission web hook to feed back into the API for read-only tracking and to reject requests exceeding quota. Michael Hrivnak expressed concern that implementing quota at the Kubernetes layer via a web hook might be too late, suggesting it should occur before Kubernetes (00:03:52).
* Quota Enforcement and Tenant Control in Cluster as a Service: Will Gordon clarified that there would be two levels of quota: one for the cluster's overall storage usage and another for developers creating Persistent Volume Claims (PVCs) within the cluster. Will Gordon and Michael Hrivnak discussed the challenge of relying on a web hook within a tenant-owned cluster, but Will Gordon noted that control plane web hooks can be used to block even cluster administrators (00:05:18). Ygal Blum asked if clusters provisioned through Ozac are intended to be a target for other services, like container or VM as a service. Eran Cohen and Michael Hrivnak confirmed that cluster as a service is intended to provide a Kubernetes API for tenants to use for their own Kubernetes tasks, with tenants in charge of maintenance (00:06:21).
* Next Steps for Cluster as a Service Storage: Michael Hrivnak concluded that they need to identify someone to investigate cluster as a service storage, pending Lars's return (00:07:31). Alona Paz suggested that since work on the VM as a service generic storage API is underway, determining if the cluster as a service API should be shared or separate is urgent. Will Gordon confirmed that an enhancement proposal for the OSAC storage provider and storage class API is being planned (00:08:35).
* Preventing Duplication in Storage API Development: Alona Paz requested a meeting between Will Gordon, Avishay Traeger, Zultan H, and Lars to ensure they avoid duplicating work on the API pieces for VM as a service and cluster as a service. Michael Hrivnak stressed the importance of mapping out the complete high-level picture of the storage architecture before detailing enhancement proposals to avoid conflicts and overlapping work (00:09:47).

* GPU Pass-Through Demo (Juan Hernández): Juan Hernández presented a recorded demo on creating a virtual machine using GPU pass-through. * Discussion on Instance Types and GPU Configuration: Oved Ourfali questioned if a generic template could support both GPU pass-through and non-pass-through, and how to handle different GPU types (00:44:57). * Improving GPU Configuration User Experience: Alona Paz expressed concern that creating a new Ansible template for every GPU type requires too much administrative work, suggesting the GPU type should potentially be part of the VM compute instance API (00:51:41). Alona Paz and Michael Hrivnak agreed that the process should be automated, making it easier for the cloud administrator, and Michael Hrivnak supported thinking about instance type definition as a selector rather than a free-form field (00:53:17). * Conclusion and Final Follow-Up Items: Due to time constraints, the two remaining planned demos were tentatively rescheduled for Wednesday. * Demo Strategy and Impact: Oved Ourfali suggests focusing demos on aspects that are more customer-visible and impactful, particularly those that could be shared externally. The discussion centers on ensuring demos, such as the VM as a Service, effectively showcase new capabilities like GPU pass-through from a user perspective. Alona Paz agrees that a full VM as a Service demo doesn't need re-recording for every new feature, but suggests periodic updates to incorporate all new features (01:00:23).
* Automation of Demos: Adrien Gentil proposes automating the demo process to streamline updates and maintenance. Oved Ourfali concurs, noting that new features might essentially be represented by a simple flag or a checkbox in the user interface.
