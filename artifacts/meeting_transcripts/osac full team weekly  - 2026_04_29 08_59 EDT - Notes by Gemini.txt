Summary
The meeting addressed workspace tooling integration and defined architectural strategies for bare metal agent provisioning and environments.

Workspace Tools and Automation
Participants discussed the OSAC workspace agentic tools and the PR notify function for tracking status. The team agreed to integrate a GitHub bot for automated Jira bug remediation.

Bare Metal Provisioning Architecture
The group aligned on using host lease Custom Resources for inventory management and bare metal lifecycle operations. This ensures consistent integration across heterogeneous backend network and provisioning systems.

Stable Integration Environment
The team determined that a dedicated single node OpenShift cluster will provide the necessary stability for user interface development. This setup isolates testing resources from shared development clusters.

Details
* Discussion of OSC Workspace and Agentic Tools: Eran Cohen presented the OSAC workspace, a meta Git repository that contains agentic skills, workflows, hooks, and architectural components (00:03:05). This workspace utilizes the GSD workflow (requiring GSD to be installed by the user) which defines a development cycle starting with discussion, research, and planning, followed by execution, verification, and shipping a pull request (PR) (00:04:20). The workspace also features Jira synchronization to transition tasks between statuses and a useful `bug fix` slash command which automates opening a Jira issue, creating a PR with the fix, writing the test, and assigning the task (00:05:44).
* Query Regarding PR Notify Tool: Lars Kellogg-Stedman inquired about the functionality of the `PR notify` tool included in the repository, as it was not documented. Elad Tabak confirmed that they were responsible for the tool, which generates a daily status of pending or stuck PRs (00:07:08). The intent is for this status to be posted on Slack to encourage reviews or provide individuals with tasks to do (00:08:05).
* PR Notify Tool Operation and Output: Elad Tabak explained that the `PR notify` tool currently posts to Slack by default, but there is a dry run option to display the output on the console (00:08:05). The tool is running on Elad Tabak's machine currently, and Lars Kellogg-Stedman noted that if run locally, it likely would not have Slack credentials (00:09:14). The purpose of the tool is to address an accumulation of unhandled PRs (00:10:14).
* Request for GitHub App for Bot Integration: Eran Cohen mentioned receiving a request to add a bot, called `autofix` or `bugs body`, to the project, similar to those on other projects. This bot would be used to automatically fix straightforward Jira bugs assigned to it, but they require a GitHub app on the organization for it to function (00:10:14). Lars Kellogg-Stedman confirmed that this could be accommodated and Eran Cohen would follow up later (00:11:27).
* Follow-up Discussion on Bare Metal Fulfillment and Agent Provisioning: Dan Manor initiated a discussion following up on the bare metal enhancement proposal, noting that the current plan does not solve the issue of automatic provisioning of servers for clusters using the discovery ISO (00:11:27). Dan Manor suggested the need for generic APIs for inventory and bare metal lifecycle management. Alona Paz emphasized that they require a cluster-as-a-service demo which necessitates agents being automatically added and synced based on an inventory before a cluster is requested (00:13:00).
* Agent Pre-Provisioning Requirements: Lars Kellogg-Stedman affirmed that automatic agent provisioning is a required part of the cluster-as-a-service plan. Alona Paz expressed confusion over how this is accomplished with the new Custom Resources (CRs) in the enhancement proposal, as the current understanding is that actions only begin when a cluster is requested, which is too late for agent provisioning (00:14:27). Tzu-Mainn Chen explained that the original design documents covered generic workflows, including templates and host leases, but noted that the Netris workflow, which requires the host to be on the correct network before provisioning, may complicate things (00:15:26).
* Planned Solution for Agent Provisioning: Alona Paz requested clarification on the plan for adding agents before cluster creation, as they need to understand how to implement this without conflicting with future API concepts (00:16:27). Tzu-Mainn Chen explained the overall bare metal fulfillment idea, which involves creating bare metal pools and host leases to expose APIs for host manipulation (00:17:21). The process relies on templates for bare metal pools and host leases that allow automated actions, such as putting the host on a specific network or provisioning an image (00:19:30).
* Triggering the Agent Provisioning Flow: Lars Kellogg-Stedman clarified that the flow is triggered when someone submits a request for a bare metal pool tagged with a specific profile (00:19:30). This mechanism allows the Cloud Service Provider (CSP) to dedicate a portion of their hardware inventory to serve as a general-purpose agent pool for cluster installation (00:20:33). Michael Hrivnak suggested that this should be automated as part of the install flow, where the system connects to the CSP inventory and provisions the requested amount (00:21:31).
* Bare Metal Inventory and Management Components: Dan Manor inquired if there would be a common interface for inventory and lifecycle management, noting that their current implementation would use different components like OpenShift Bare Metal Operator. Lars Kellogg-Stedman explained that the common interface is the `host lease` CR, which is used for allocating hosts (00:24:12). Tzu-Mainn Chen clarified that the Bare Metal Fulfillment Operator has a Host Inventory Controller that takes the `host lease` CR, talks to the backend inventory (which is pluggable), and assigns a host to the lease (00:25:21).
* CSP Role in Agent Pre-Provisioning: Oved Ourfali summarized the CSP's role: they need to create the bare metal inventory and a bare metal pool CR specifying the number of agents and the ISO image for pre-provisioning (00:26:21). The Bare Metal Pool Operator then creates host CRs based on the available inventory. Lars Kellogg-Stedman added that the assumption is that the inventory already exists (e.g., Netbox, OpenStack, or a file), and the project must provide credentials for an inventory connection (00:27:33).
* Current Backend and Networking Integration: The current inventory backend being developed is OpenStack, which uses the Ironic API for both inventory and bare metal management (00:28:47). Oved Ourfali raised the question of how to handle networking, specifically since Dan Manor is working with Netris for network isolation. Lars Kellogg-Stedman confirmed the goal is to handle network isolation through the OSAC network API, with backends supporting the target technology (e.g., Netris or Ironic/Neutron) (00:30:16).
* Concerns Over Operator Implementation for Backends: Rom Freiman voiced strong concerns that implementing separate operators for every small integration, such as for different inventory backends, would be unacceptable due to the overhead of productization and resource consumption (00:39:58). Lars Kellogg-Stedman defended the current design, stating it follows Kubernetes conventions, like storage classes, which use different operators to support different types of backend hardware (00:40:57). Rom Freiman argued that a single operator could support multiple backends, which is the model used for the Inventory Operator (00:42:03).
* Decision on Starting Agent Provisioning and Documentation: Alona Paz suggested that since the new template integration is not yet in place, the fastest path forward is to write a playbook similar to the existing ESI playbook for provisioning agents (00:49:57). Lars Kellogg-Stedman agreed that this is the best path if the goal is to have a working solution within a week (00:50:59). Alona Paz requested that Tzu-Mainn Chen and Dan Manor meet to create a sequence diagram detailing the full cluster-as-a-service flow using the new APIs, including a file for inventory, OpenShift Bare Metal Operator for provisioning, and Netris for networking (00:52:29).
* Integration Environment Stability for UI Team: Adrien Gentil highlighted the need for a stable and isolated integration environment for the UI team's development work, as the current development cluster is unstable due to shared, cluster-wide resources being overwritten (00:54:25). Lars Kellogg-Stedman agreed on the need for a stable deployment for testing (00:55:49). Orran Krieger suggested that while a virtual cluster might be sufficient, it should ideally run on a separate, dedicated environment to ensure stability (00:56:50).
* Proposed Solution for Stable Integration Environment: Lars Kellogg-Stedman proposed setting up a single node OpenShift cluster on a large machine (e.g., an 830) with virtualization enabled, to be used as the stable target. Orran Krieger agreed that this approach is in scope, provided it does not use GPUs (00:57:42). Adrien Gentil was asked to create a ticket for this task and assign it to Lars Kellogg-Stedman (00:58:33).
