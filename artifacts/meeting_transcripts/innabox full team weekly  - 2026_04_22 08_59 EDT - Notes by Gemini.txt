Summary
Meeting discussions addressed metering requirements and storage proposals for OSAC via architectural review and roadmap planning.

Metering and Event Requirements
Participants summarized MOC metering documents, emphasizing metadata capture and event-based tracking for short-lived resources. Integrating these requirements into the OSAC project remains a primary objective.

Storage Proposal and Quotas
The team decided to mandate quota enforcement at the storage provider level for cluster and bare metal services. This approach pushes multi-tenancy logic directly into storage mechanisms.

Implementation Roadmap and Inventory
Developers will utilize Ansible to orchestrate storage provisioning before building dedicated Storage API endpoints. The group also identified reviewers for the bare metal inventory interface pull request.

* [Will Gordon] Implement Storage: Create initial Ansible playbooks for storage provider and tier configuration. Automate consumer side storage provisioning for VM as a Service and Cluster as a Service.
* [Trey West] Review Inventory PR: Examine the inventory host controller PR interface definition. Provide feedback to Austin Jamias regarding necessary changes for backend implementation.
* [Austin Jamias] Post PR Link: Add the inventory host controller PR link to the meeting agenda document.




Details
* Initial Greetings and Location Discussion: The meeting began with greetings among the participants. Lars Kellogg-Stedman asked about Will Gordon's location, and Oved Ourfali indicated that they were assigned to the Raleigh office, though the exact location was not remembered. Orran Krieger shared an anecdote about their new cabin in Maine having commercial flights only to Florida or Raleigh, North Carolina, but not Boston (00:00:00).
* Agenda Item Exclusion (Storage Proposal): Lars Kellogg-Stedman noted that the first agenda item, Will Gordon's storage proposal, would be skipped due to Will's initial absence and the fact that several attendees were out on holiday. The intent was to bring up issues from a previous discussion with the larger group (00:04:38).
* Metering Document Discussion Overview: The second agenda item focused on metering, specifically the broad discussion within a dedicated document and a MOC-specific document detailing requirements for that environment (00:04:38). Lars Kellogg-Stedman sought to determine if participants wanted an active discussion or just awareness of the existing documents (00:06:00).
* MOC Metering Document Summary: Orran Krieger suggested a summary of the MOC metering document to provide context to the larger group (00:06:00). Thorsten Schwesig provided a summary, noting that the MOC document builds upon Michael Hrivnak's proposed document by adding MOC-specific personas, such as the student and PI. The MOC document emphasizes the need for metadata from inventory or events at the start to ensure fair pricing and billing later (00:06:58).
* Metering Focus on Events and Metadata: Thorsten Schwesig explained that metering must include metadata (like who used a GPU or from which pool) so the customer can determine pricing. The MOC document also stresses that focusing on events, rather than just observability, is crucial for capturing short-lived resources, especially those under 30 seconds (00:08:18).
* Event Broker Requirement for OSAC: Orran Krieger advocated for getting the MOC requirements out to ensure no special code is needed in the MOC, aiming to incorporate requirements into the OSAC project (00:09:30). Christopher Tate stated that while a solution is not needed, a message broker that allows multiple services to subscribe to published events would be beneficial (00:10:32).
* Storage Proposal: Initial Requirements: Upon Will Gordon's arrival, the discussion shifted to their storage proposal, which focused on two initial requirements (00:10:32). The first requirement is a resource (a credentials holder wrapper) for storage provisioning, allowing a CSP admin to set up provider access information. The second requirement is a catalog resource, proposing a "storage tier" instead of the previously suggested "storage class," to list available storage options (00:11:39).
* Storage Catalog Integration Discussion: The discussion explored where the storage catalog resource should live, noting that OSAC has a template catalog used for virtual machines and cluster as a service (00:12:59). Michael Hrivnak and Lars Kellogg-Stedman suggested a dedicated Storage API endpoint, similar to the Network API, rather than integrating into the existing template catalog. Will Gordon noted that storage is often a just-in-time provisioned resource, although Lars Kellogg-Stedman countered that provisioning clusters or image registries can require immediate storage attachment (00:13:55).
* Quota Enforcement Mechanisms: Will Gordon outlined three proposals for quota enforcement: using a CSI proxy, employing a validating admission webhook, or relying on eventual consistency with the storage provider handling its own quotas (00:15:54). Lars Kellogg-Stedman raised feasibility concerns with the first two options in a cluster-as-a-service context, as a tenant could potentially disable or bypass them (00:17:12).
* Quota Enforcement on Hosted Control Plane: Will Gordon detailed a potential solution for hosted control plane (HCP) environments by running a validating admission web hook service on the control plane side. This approach involves either requesting a change in HCP's hard-coded list of resources or running a separate operator to reconcile the protected resources if a cluster administrator attempts to delete them (00:18:14).
* Feasibility of HCP-Only Solution: Oved Ourfali and Michael Hrivnak expressed concern about adopting a solution that only applies to HCP, noting that dedicated or standalone control planes might be preferred in sovereign environments or for single-node open shift (00:20:41). Michael Hrivnak also stressed that quota enforcement needs to be implemented at all layers, especially since the CSI driver holds the necessary credentials on the worker nodes (00:21:46).
* Delegating Quota Enforcement to Storage Provider: The group discussed approaching virtualization and cluster/bare metal services separately for quota enforcement. Lars Kellogg-Stedman suggested that quota enforcement for clusters or bare metal must occur at the storage provider level, making it a requirement for fully supported storage backends (00:23:00). Michael Hrivnak agreed with this direction, mentioning that leveraging soft and hard limits supported by providers like Vast seems like a good starting point (00:24:07).
* Multi-Tenancy and Storage Quotas: Implementing quotas on the storage back end suggests pushing multi-tenancy logic into the storage mechanism itself, which is supported by Michael Hrivnak (00:24:07). This allows storage providers to manage tenant-specific configurations such as encryption and quality of service. Orran Krieger agreed with leveraging provider capabilities but insisted on having an overarching system within OSAC to track quotas across different storage types and tiers (00:25:25) (00:29:07).
* Storage API and Provider Configuration: Will Gordon proposed having a Storage API endpoint for CSPs to manage storage providers and a "storage tier" resource that tenants can read, with Ansible reconciling quotas to the provider (00:31:30). Michael Hrivnak suggested that OSAC might be able to function without new API endpoints initially, by leveraging Ansible for provider configuration and integrating it into the workflow via environment variables or secrets (00:37:32).
* Storage Implementation Roadmap: Will Gordon clarified Michael Hrivnak's suggestion, which is to first use Ansible to orchestrate basic setup and provisioning of storage (the consumer side), and then improve the CSP experience (the provider management side) later by adding APIs (00:39:57). The Ansible implementation would require a well-defined interface, similar to networking, to allow for swapping different provider drivers (00:42:25).
* Storage Abstraction and Tenant Interaction: Ameya Sathe inquired whether the abstraction would cover block, object, and file storage, and whether it would rely on OpenShift CSI drivers or Ansible. The consensus was on using Ansible to create and configure the clusters, including CSI driver installation and storage class creation (00:43:30). The discussion confirmed the intention is for OSAC to abstract the storage tier (e.g., "fast block," "slow NFS") which tenants can request, regardless of the underlying provider (00:52:09).
* Bare Metal Inventory PR Review: Lars Kellogg-Stedman moved the discussion to the final agenda item concerning bare metal inventory. Austin Jamias requested feedback on a PR for the inventory host controller, which includes an interface for inventory backends. Trey West, who is working on Carbide support, was identified as a candidate to review the interface and provide feedback (00:53:28).
