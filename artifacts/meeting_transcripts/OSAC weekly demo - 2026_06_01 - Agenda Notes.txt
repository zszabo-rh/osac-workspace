Purpose
Delve deeper into technical questions around bare metal fulfillment[ae][af][ag][ah].[ai][aj]
Context
Today in the MOC, we use ESI[ak][al] for bare metal fulfillment capabilities. ESI is simply OpenStack (primarily Ironic, Neutron, and Keystone) configured for Ironic multi-tenancy and Neutron ML2-Ansible networking. Using the OpenStack CLI allows for the following:
   * Administrators can lease nodes to tenant projects, giving them exclusive access
   * Tenants can configure node networking with public and private Neutron networks
   * Tenants can control node power state
   * Tenants can access a node’s serial console
The MOC requires all of these features in the Cloud-in-a-Box solution. Currently, we are unaware of any alternative to ESI that provides them; as a result, in the short-term we are planning on continuing to use ESI for bare metal fulfillment.
This decision has consequences for cluster fulfillment as well, since we would like the bare metal resources that ACM uses to be retrieved from the same source as that used for bare metal fulfillment; ACM is effectively another tenant project of ESI. This places constraints on the choice between using ACM BareMetalHosts versus Agents: the former requires BMC access, and we do not wish to expose direct access; at best we would stand up a BMC proxy. In order to avoid this added infrastructure requirement, we have currently decided to use Agents[am][an].

[z]Right now we simply select and label agents CRs; the hosts behind those agents are never marked. This change selects those hosts first through a bare metal pool CR to allocate hosts pre-provisioned as agents, and then labels the agents corresponding to those hosts.


Once it's in place, work can begin on support for Carbide in the host inventory operator (Austin has already asked Trey to look at the interface he's using for OpenStack to see if it also works for Carbide), as well as for a Carbide host management operator.


This work also is the "spine" that allows us to work on the missing bits simultaneously afterwards.
[aa]I see, thanks. Does it mean that agents will be provisioned only after a baremetal pool will allocate them? Meaning only when a cluster is requested?
[ab]Nope; part of the BMaaS design includes the idea of being able to identify hosts that are free, but which are currently pre-provisioned as agents. So when CaaS calls BMaaS, part of the parameters sent include asking for hosts that are already agents. Those hosts are assigned, and then CaaS can use that information to identify the Agent CRs to use, and the rest of the workflow is the same.


The second part of this - and the part which is further away - is the CSP workflow of maintaining an active pool of agents. After identifying the number of agents desired, this workflow would call BMaaS to identify hosts that are free (and not provisioned as agents), and then provision them as agents.


For the MOC 2.0 we've considered the first part more important (for our initial deadline of June 1st) because it is needed to track host allocation, and because it's a fundamental architecture change to CaaS. The second part is something we can use our current ad hoc playbooks for in our initial evaluation deployment. That being said, we do consider it important, and after our initial May 4th demo we'd definitely be continuing to implement the features needed for it (specifically image-based provisioning for hosts, and the capability to specify workflows for a host through an AAP template).
