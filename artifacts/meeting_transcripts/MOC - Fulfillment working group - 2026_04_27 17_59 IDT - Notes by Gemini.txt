Summary
Quotas and metering direction discussed OpenMeter implementation with a reliable event source required for billing accuracy and Carbide integration demonstrated cluster lifecycle.

Quotas and Metering Direction
The team debated OpenMeter versus Red Hat cost management as the preferred solution for quotas and metering, noting that the existing event source for billing is unreliable. A reliable event source is needed, possibly requiring a cursor implementation, as billing should be based on reservation or utilization but requires accurate usage tracking.

Cluster as a Service Workflow
Discussion centered on completing the Cluster as a Service workflow by automating the bare metal server registration via a live ISO to create assistant installer agents for a hot server pool. The current process relies on a manual Ansible playbook, and integration with the OSAC installer and Enclave is required for host configuration.

Carbide and Networking Demos
A demo showcased the Carbide/Nico cluster creation and deletion flow, utilizing VPCs and instance types, though the current implementation is specific to an NVIDIA setup and has multi-tenancy constraints. A separate demo confirmed VM virtual networking isolation by successfully restricting connectivity between VMs in different security groups and virtual networks.

Details
* Quotas and Metering Direction: Discussion began regarding the direction for implementing quotas and metering, with an existing document suggesting OpenMeter as an implementation choice for metering, while noting Red Hat is also advancing their on-prem cost management solution to include billing. A decision is needed to choose between OpenMeter or the Red Hat cost management system, as the chosen direction will define necessary infrastructure for quotas and events (00:04:55). It was highlighted that metering and quotas are tightly coupled, as usage tracking (metering) is necessary to enforce quotas (00:12:54).[a][b][c]
* Event Source for Billing: A key topic was the need for an event source, as the MOC currently bases billing on event information rather than metrics interfaces like Prometheus (00:01:31). There is already a gRPC streaming service endpoint for events in the fulfillment service that publishes changes in the API, though this implementation is unreliable since it uses PostgreSQL listen and notify, meaning events are missed if the client is not present when they are generated (00:02:58). To support robust billing systems, a more reliable solution, such as implementing a cursor for reconnection, is required (00:04:02).
* Metrics-Based vs. Event-Based Metering: There is a distinction between metrics-based and event-based metering; OpenMeter uses events, while the Red Hat cost management approach uses metrics. The choice impacts the accuracy and source of truth for quotas, suggesting a unified approach is needed to define requirements for events and quotas (00:04:55). Lars Kellogg-Stedman raised that OpenMeter could be a data source for an organization's pre-existing billing logic, making appropriate third-party integration endpoints necessary (00:06:10).
* Billing Based on Reservation vs. Utilization: The discussion clarified that billing can be based on resource reservation (e.g., quotas) or utilization (e.g., CPU usage) (00:07:22). Reservation-based billing, such as for storage, makes sense because a reservation prevents others from using the resource, but VM quotas might not prevent others unless the entire quota is consumed (00:08:42). The final decision rests on whether the system should bill based on resource consumption or reservation (00:09:42).
* Event Persistence and Auditing: The possibility of using a live event endpoint with resumability was discussed, which would allow external systems to handle long-term event persistence outside of OSAC (00:10:37). However, Will Gordon argued that a cloud provider, such as OSAC, should be expected to persist billing information long-term for the CSP admin, making the proposal for short-term persistence "crazy talk". This led to questions about how closely OSAC should be tied to billing, acknowledging it is a difficult decision without a current customer (00:11:53).
* Image Management Update: Avishay Traeger provided an update on image management, noting that several epics and an enhancement proposal have been opened and reviewed. The goal is to move beyond the current assumption—where an admin uploads an image to a registry and references it in Ansible—to allow API access for uploading images, referencing them, and saving VMs to an image for backup purposes (00:13:49).
* Completing Cluster as a Service Workflow: Dan Manor introduced the discussion on completing the Cluster as a Service workflow from the infra admin’s perspective to ensure a fully automated end-to-end flow (00:15:32). The flow requires deploying VMs with a live ISO to create assistant installer agents for a hot pool of servers, which is necessary because Cluster as a Service currently relies on pre-requisite steps like configuring a network backend (OpenStack or Netris) and registering servers (00:16:41).
* Automation of Bare Metal Server Registration: The process of booting servers with the discovery ISO is automated on the OpenStack side using an existing Ansible playbook to register servers as agents, though this is not yet exposed via a command-line tool (00:17:56). There is currently no automated hardware monitoring system to eject degraded servers from the hot pool, although that is a goal (00:18:54). Oved Ourfali suggested that the Cluster as a Service flow needs to integrate with the OSAC installer and Enclave, which handles management cluster provisioning and host configuration (00:20:06).
* Bare Metal Host Inventory and Agents: The current architecture assumes an external inventory for bare metal as a service, and hosts are pre-provisioned and labeled as agents to be queried as free hosts (00:25:22). Cluster as a Service works with these agents, and the agents are registered using a separate workflow, currently a manual playbook, with future plans to automate it (00:26:23). The single source of truth for these agents is the external inventory, which is associated with ACM through a playbook that labels the agents for proper identification (00:27:20).
* Carbide/Nico Integration Demo (Cluster Creation): Trey West demonstrated the OSAC integration with Carbide, which has been renamed Nico. The cluster creation flow leverages the concept of VPCs for tenant separation and instance types (equivalent to OSAC's host classes) for hardware definition, but unlike ESI, agents are registered at fulfillment time, not pre-registered (00:30:29). External access is handled by Metal LB and BGP due to networking constraints in Carbide (00:32:52).
* Carbide/Nico Integration Demo (Deletion and Considerations): The deletion flow reverses creation, tearing down the hosted cluster, cleaning up external access and agents, and optionally deleting the VPCs and prefixes if they were created during the process (00:32:52). High-importance considerations include the current implementation being specific to an NVIDIA setup, potential conflicts between OSAC and Nico multi-tenancy, and missing API capabilities that prevent pre-registering agents due to VPC constraints (00:34:06).
* Carbide/Nico Integration Demo (Configuration and Scaling): The demo showed the use of a custom cluster template specifying host classes, which map to Nico instance types, and parameters for VPC management (00:37:34). The backend creates the requested instances and a VPC prefix, which is a required component for IP block management (00:41:52). The demo also confirmed that scaling up and down is handled by provisioning and terminating instances on the backend, with BGP peers created for new nodes (00:45:59).
* Carbide/Nico Networking Details: The nodes in the installed cluster receive their IPs via DHCP from Carbide, which includes IP management, DHCP, and DNS capabilities for the underlay network, using VXLAN for the overlay. Trey West was unsure about the underlying physical architecture (e.g., spine leaf) or how it relates to potential static networking requirements observed in other backends like Netris (00:47:57).
* VM Virtual Networking and Security Groups Demo: Adrien Gentil provided a recorded demo focusing on VM creation within isolated virtual networks and subnets, and the usage of security groups. The demo showed a virtual network being created with a specific network class and CIDR, followed by a subnet creation (00:51:41). Security groups were used to initially deny all ingress and later patched to open port 80, enabling connectivity between a bastion host and a web server in the same virtual network (00:52:56).
* Virtual Network Isolation Demonstration: The final part of Adrien Gentil's demo showed that when a third VM was created in a separate, new virtual network, it was completely isolated from the first two VMs, proving the network isolation functionality within the same tenant (00:56:03). The inability to route multiple subnets within the same virtual network is a current implementation limitation awaiting a future OpenShift networking improvement (00:52:56).
* Discussion of Demo Subnet Configuration: Adrien Gentil presented a console view of a machine booting, confirming that the machine uses a different subnet (220.210.200) from the one previously expected. Eran Cohen suggested that a better demonstration might have involved showing machines on the same subnet that were unable to communicate, but Adrien Gentil confirmed that the current setup was not hosted (00:57:27).
* Upcoming Demonstration Availability: Oved Ourfali inquired about another planned demonstration, but Adrien Gentil explained they did not have time to record it. The necessary infrastructure was not available due to issues with CN from the previous weekend (00:57:27).
* Schedule for Remaining Discussion Items: Oved Ourfali introduced the two remaining discussion items and asked if attendees could stay on the call for a few more minutes. Lars Kellogg-Stedman suggested either setting a hard time limit for the remaining items or punting the discussion to the next meeting on Wednesday or Monday (00:57:27).
* Meeting Conclusion and Next Steps: Oved Ourfali concluded the current meeting due to time constraints, noting that they had just one minute left. They thanked everyone for joining and for the demonstrations, suggesting that further discussions should continue offline, with the next scheduled meeting being on Wednesday (00:57:27).
